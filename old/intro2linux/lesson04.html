Logging In

The first thing you will need to do is log in. This is quite simple, simply enter your username (root), press enter, enter the password at the password prompt and press enter. Congratulations, you just logged into your system as root. If this failed, make sure the username is in LOWER case and the password has been entered correctly.

Learning about commands

Linux is extremely well documented, you should not expect to become a Linux expert straight away, while it maybe frustrating you will benefit from spending the time to read the manual pages. Linux has a command called man. Think of man as an on-line quick reference guide for almost every application, utility and command on your system. The man pages work just liked the unix command less. To quit out of man pages press q, and use the arrow keys for navigation. Man is short for manual. You should use the man command to learn about commands you are not sure about.


Red Hat Directory Structure

Linux distributions and other UNIX type operating systems such as Solaris have a particular directory structure. You may notice commands that work the same in Linux as they do in other UNIX operating systems, with minor differences. The one key difference between one Linux distribution and the next is where files are located. The Red Hat Linux directory structure is used in many distributions, including Caldera OpenLinux. The directory structure and commands are also similar with other operating systems under the BSD tree, including NetBSD and FreeBSD.

Above is the basic directory map, we will now explain the function of each directory.


/

Root (or slash) is the top level directory, there are no other directories above this. This is called the root directory.


dev/

/dev (sort for device) contains special device files. There are two types of device files (character and block). Block devices are generally hard disk type devices, while character devices are generally mice, video capture cards etc. Each device is given a major and minor number, this is how the Linux kernel knows how to address a particular device. You can create your own devices with the mknod command.




Boot/

This is the boot directory, this is where you store your linux kernel (vmlinuz) and system map files. This directory is discussed in detail in Appendix A.


var/

The var directory contains logs, mail spools, news spools, print spools etc.


usr/

The usr/ (USER) directory contains user run applications, utilities and configuration options for those applications. This directory has its own

directory structure:


usr/local/ Third Party Applications are installed here

usr/lib/ User Libraries

usr/sbin/ User System Binaries (utilities)

usr/bin/ User Binaries (applications)

usr/etc/ User Configuration Files

usr/X11R6/ X-Windows

usr/man/ User manual pages

usr/include/ User include (.h C header files)

usr/src/ User source code


The usr/local directory once again mimics the usr level directory with bin/sbin/ etc/ lib/ man/ include/ and some other sub directories. You can navigate between these directories with the cd command


Example: cd /usr/local

cd /usr/bin


If you download a new third party application, such as the AbiWord free word processor, you would install this package in the /usr/local/ tree, although you could install it anywhere you like, the standard is to install such third party applications in /usr/local/. The idea being, that it makes it easier to locate third party applications, as well as to backup, remove or restore such applications.



mnt/

mnt (mount) is a special directory. This is where you can add file systems into your linux system. Say for example you have an IDE carom drive on (/dev/hdd). Since the cd-rom cannot be partitioned into separate partitions, the device does not have any partition numbers, it is simply referred to as /dev/hdd. If you place a cd into the drive, you can mount that drive in /mnt/cdrom. To access the directories and files on that cd you simply cd to /mnt/cdrom and use ls, cd and other system level commands to browse the cd. You can mount caroms, zip drives, hard disks and network shares (NFS and Samba) with the mount command.


Example: mount /dev/hdd /mnt/cdrom -t iso9660


In the above example, the mount command is issued on our carom drive in

/dev/hdd, the /mnt/cdrom is an empty directory that exists, it is called

the mount-point. This can be any empty directory anywhere on the system,

generally you mount drives in /mnt unless you have multiple drives and

wish to use extra drives as system directories such as /usr/local. The

-t flag specifies the type of partition. Since this is a cd-rom it

will be in the ISO 9660 standard, so we tell mount it is such.




home/

This is the home directory. This is where user's directories are created. Each user you create (username + password) is given a home directory. This is where they can store files. On some systems you may wish to put a quota in place to prevent users store large amounts of data in their home directories and exceeding your limited disk space.


lib/

This is the system library directory. In here you will find system level libraries. You really shouldn't need to enter this directory unless you are upgrading system libraries. This is where libraries such as libc6 are stored.


bin/

This is the top level program directory. This is where non-root files are stored. Generally ls, cat and other user accessed system level programs are stored.


sbin/

This is where system programs are stored. These are programs such as mount and ifconfig are stored. These are programs which generally require root or root privileges to execute correctly. Generally these are programs that require some kind of device access, such as exclusive access to your hard disk, keyboard, monitor etc.


etc/

etc (pronounced like etc) is where the system level configuration files are stored. There are some key files in here which you should pay attention to:


password Where the system usernames and home directories are stored

shadow Where the encrypted passwords are stored

group Where system groups are listed

resolv.conf Lists the systems DNS servers

HOSTNAME The name of the system

smb.conf The samba (windows file/print sharing) configuration

inetd.conf Basic tcp services configuration (telnet etc)

hosts.allow Which remote hosts/ips are allowed access to which services

hosts.deny Which remote hosts/ips are denied access to which services

hosts List of hostnames and the IPs that match

issue Message displayed before login prompt

issue.net same as issue

motd Message of the Day (displayed to users when they login)

stab Lists which partitions to mount on boot and where

services Lists which services run on which tcp/udp ports

rc.d Startup scripts


proc/

This is the directory where the system stores information about itself.

For example if you cat /proc/cpuinfo you will get information of your processor. Try using cat to read information stored in files in /proc.

The /proc/interrupts file can be used to check for IRQ conflicts.


tmp/

This is the system temp directory. This is where all temporary files are written to. Generally all users can write to /tmp.


root/

This is the root (super user/administrator) account's home directory.


lost+found/

This directory is found inside all ext2 (Linux native) partitions. Whenever a problem occurs with the file system, this is where recovered information is stored. Files stored in here are generally listed by node. See the e2fschcommand for further information on inodes.


Linux Shell


When you logged into your Linux system as root, the prompt you were given is called a BASH prompt. BASH is the name of your shell. A shell is a program that provides the user with a means to execute commands, and view the file system. There are many different shells available, bash is the default for Red Hat Linux. The other popular shell is tcsh. This is an extended version of the C shell, named so as it is based upon the C programming language. The tcsh shell is somewhat easier to use than bash.


Another feature of these shells is what is known as tab completion. This allows you to press the tab key to allow the shell to complete the name of afile or program. For example, if you wanted to edit a file called mybusinessplan.txt, you could simply type pico -w mybus TAB and the shell would complete the rest of the name for you.

Users and Groups


Now that you are familiar with the file system, we will discuss file permissions. The username is provided as a human readable form, Linux itself uses UID and GID to determine who the owner of a file is.


UID

User ID, this is a unique number assigned to a particular user. Files that this user creates are stored with that ID number. Root always has a UID of 0.


GID

Group ID, like User Ids are unique numbers assigned to groups of users. Groups allow administrators to allow a group of users access to a particular system resource. Users can belong to multiple groups, but always have a default group.


Passwd File

The password file stored in /etc/passwd contains the mapping of UIDs to usernames.


joe:x:540:500:Joe User:/home/joe


The first field in the password file is the human readable username. The second field is a legacy field, this used to be where the encrypted password was stored, the x denotes that the encrypted password is stored in the shadow file. The third field is the UID, this is unique and duplicates cannot exist. The forth field is the default group ID, generally on Red Hat systems, group 500 is the default users group. The fifth field contains an ID string for that particular user, generally this is the users name. The final field contains the path of that users home directory. The home directory must be owned by UID 540.



Groups

Aside from the default user group, which doesn't have a list of users that belong to it in the /etc/group file. If there is a need to provide one or more users access to a resource, such as access to a documents directory for the sales department, you would create a sales group. Add the users who should have access to that group and then set the group permissions on the directory to the sales group.


In the /etc/group file you would add:


sales:x:600:joe,jim,bob,tim,emily,sarah


This adds the users joe, jim, bob, tim, emily and sarah to the sales group.

The /usr/local/doc/sales is the directory you want to give them access to.


drwxr-x--- 24 root root sales


To obtain this output you need to use ls -la in the /usr/local/doc directory. The first section of letters, indicates the permissions set on the file.


The first character, in this case d indicates that this is a directory, regular files have a - in this field. The next three characters, rwx provide read, write and execute permissions for the owner of the directory. The second triplet indicates that the group has read and execute access, but cannot write. The final triplet has no letters, this triplet indicates the permissions for the rest of the system, here the rest of the system has no permissions to this directory.


Since the owner is root and the group is root, the sales group doesn't have access. You can change the group ownership by using the chgrp command.


chgrp -R sales /usr/local/doc/sales


The above command, with the recursive -R option, changes the group ownership in the sales directory and below from root to the group sales. Once this command has been completed, the users in the sales group have read and execute permissions.


File Permissions


File permissions are displayed with the -la option in ls. The permissions are grouped into three triplets.


1222333444 X owner group size date filename


The above is a key for the output you would expect to get from ls.


1 = File Identifier (- = regular file, d = directory, l = link, b/c = device)


2 = Owner permissions

3 = Group permissions

4 = Other permissions


owner = the username or UID of the owner

group = the groupname or GID of the group

size = space used by the file or directory in bytes

date = Last modified date/time

filename = the name of the file or directory


The owner, group and other permissions each have three entries, first for read permissions, second for write permissions and the third for execute permissions.


You can change these permissions with the chmod command.


chmod oug+rwx file

chmod oug-rwx file


The first example gives (other, user and group) read, write and execute permissions to file. The second example removes (other, user and group) read, write and execute permissions. You can also use what is known as a UMASK option, this is an advanced method of changing file permissions. Your instructor may choose to provide UMASK examples, see man chmod for information on UMASKs.


Creating User Accounts


The root account is the only user account on the system after the installation. However, you cannot log in remotely as root. This is mainly a security feature of Linux. In order to be able to log in to the system remotely, as well as to administer the system, we suggest you create your own account. This can be anything you want, as long as it is less than 8 characters long.


To create the account simply use the adduser script or you can use the useradd command. Example:


adduser username


Simply, replace username above with your desired username.

Once created, you will also need to add your password. To do this simply use the password command. Example:


passwd username


This will prompt you for a password, and then confirmation of the password. Remember the password advice we gave in the last lesson!


Once the password has been set you can test the account. Do you remember the virtual consoles during the Red Hat installation? The same virtual consoles exist in Linux itself, you have six virtual consoles. You can access each by pressing ALT+F? where ? is replaced with a number from 1 to 6. You are currently on the first virtual console (ALT+F1). There is a seventh virtual console on ALT+F7 however this virtual console is for X-Windows.


Press ALT+F2 and log in with your created account.

You can now press ALT+F1 to return to your root access account.













Switching to root


If you are already logged into a Linux system, and you know the root password, you can switch to the root user without having to log in. To do this you need to use the command su. Example:


bash~> su -

Password:

bash#>


In the above example we have sued to root. You can see the prompt change from ~ to # indicates you have root access. The command whoami will also report that you are now root. If you enter a username after su - you can become that user if you know their password, or if you are root.
